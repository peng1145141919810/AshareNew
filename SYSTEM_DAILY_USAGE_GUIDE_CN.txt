系统日常使用说明（通俗版）

这套系统不要把它理解成“一个脚本”。
更准确地说，它现在是四层：

1. 研究层
   负责看消息、看行业、看市场状态、看技术面、跑 V5 研究，最后产出“目标组合”。

2. 发布层
   负责把“目标组合”冻结成正式版本 release。
   执行层只认 release，不直接认研究过程里的临时文件。

3. 执行层
   负责读取最新 release，再结合真实账户、真实持仓、真实委托、真实成交去做执行。

4. 时钟 + 安全 + OMS
   时钟负责到点唤起执行。
   安全层负责决定现在能不能动。
   OMS 负责真实账户 / 订单 / 成交 / 实际仓位的最终真相。


一、你平时最应该记住的 3 个入口

1. 正式入口
   F:\quant_data\Ashare\launch_canonical.py

   这是你平时最应该从这里启动的入口。
   它会先做轻量检查，再转去真正的业务入口。

2. 业务根入口
   F:\quant_data\Ashare\main_research_runner.py

   这是系统真正分发各种 mode 的地方。
   一般只有在你明确知道自己在做什么时，才直接用它。

3. 常驻时钟入口
   F:\quant_data\Ashare\trade_clock_service.py

   这是一直挂着的轻进程。
   它不做重研究，只负责盯时间、看 release、看安全状态，到点后再唤起 execution_only。


二、最核心的一句话

日常正式使用时，推荐把系统拆成三步：

1. research_only
   夜里跑研究，产出最新目标组合

2. release_only
   把目标组合冻结成正式 release

3. execution_only
   到交易时间再根据最新 release 去执行

也就是说：

研究负责“想”
发布负责“定稿”
执行负责“做”

不要再把它理解成“一个大脚本一次跑完全部事情”。


三、你平时最常用的 profile

1. quick_test
   轻量调试档。
   不是秒级冒烟测试。
   它仍然会走真实链路，只是比 overnight 轻。

2. overnight
   夜间全量研究档。
   用于更完整、更重的正式研究。


四、你平时最常用的 mode

1. research_only
   用途：
   跑正式研究链。

   它会做的大致事情：
   - 抓市场和事件
   - 事件抽取
   - 三机制行业路由
   - 市场状态 / 资金面总阀门
   - 技术确认
   - V6 research brief
   - V5 研究执行
   - 组合建议
   - release 发布

   适用场景：
   - 每天收盘后或夜里跑
   - 为第二天准备目标组合

2. release_only
   用途：
   不重跑研究，只把最新组合建议重新发布成正式 release。

   适用场景：
   - 你已经有最新的组合建议
   - 只是想重新生成正式 release

3. execution_only
   用途：
   只做执行，不重跑研究。

   它会做的大致事情：
   - 读取最新 release
   - 看今天是不是交易日
   - 看现在是不是允许的交易时段
   - 看 safety 是否允许
   - 看 precision_trade 是否打开
   - 进入 OMS
   - 由 OMS 结合真实 broker truth 决定如何继续或执行

4. integrated_supervisor
   用途：
   老的综合联调模式。

   说明：
   还能用，但不推荐当成精准撮合的正式生产路径。
   更适合联调和历史兼容。

5. full_cycle
   用途：
   看研究前半链是否通。

   大致到桥接产物为止，不碰执行。

6. ingest_only / extract_only / gap_only / industry_router_only / plan_only / bridge_only
   用途：
   分段查问题。

   什么时候用：
   - 只想查抓取：ingest_only
   - 只想查抽取：extract_only
   - 只想查缺口：gap_only
   - 只想查三机制：industry_router_only
   - 只想查研究计划：plan_only
   - 只想查桥接：bridge_only

7. oms_validate
   用途：
   不碰真实研究长链，只做 OMS 的轻量 synthetic/replay 验证。

   什么时候用：
   - 你改了执行侧、OMS、连续性、手工干预逻辑后
   - 想快速检查 OMS 状态迁移有没有坏


五、你平时最常用的执行模式

系统现在有两种 execution_mode：

1. simulation
   模拟撮合
   适合联调和验证执行逻辑

2. precision
   精准撮合
   更接近真实交易时间约束

另外还有一个非常重要的开关：

precision_trade

1. on
   精准模式允许真正进入执行动作

2. off
   精准模式下只做巡检、写状态、写日志，不真正提交执行

这意味着：
你可以让系统一直在线，但今天不想交易时，只要把 precision_trade 关掉即可。

九、你每天最推荐的操作节奏

方案 A：手动日常使用

1. 收盘后
   跑 research_only

2. 研究完成后
   看最新组合和 release 是否正常

3. 次日开盘前
   先跑 execution_only --gate-only

2. 研究后自动或手动 release_only
3. 白天让 trade_clock_service 常驻
4. 常驻进程到点只唤起 execution_only

十一、如果系统“没动静”，先按这个顺序排查

1. 看是不是根本没 release
   看 latest_release.json

2. 看是不是不在交易窗口
   看 execution_only --gate-only 输出

3. 看是不是 precision_trade 关着
   看 local_settings.py 或命令行参数

4. 看是不是 safety 卡住了
   看 system_safety_state.json

5. 看是不是 OMS 里已有未完成 intent / order
   看 latest_open_intents.json
   看 latest_intent_continuity_report.json

6. 看是不是被人工冻结了
   看两个地方：
   - F:\quant_data\Ashare\data\trade_clock\manual_overrides.json
   - F:\quant_data\Ashare\data\live_execution_bridge\oms_v1\manual_overrides.json


十二、这两个 manual_overrides 不一样

1. trade_clock manual_overrides
   路径：
   F:\quant_data\Ashare\data\trade_clock\manual_overrides.json

   作用：
   控全局安全和时钟层。
   比如今天全部停手、只减仓等。

2. OMS manual_overrides
   路径：
   F:\quant_data\Ashare\data\live_execution_bridge\oms_v1\manual_overrides.json

   作用：
   控某个 symbol / intent / 连续性修复。
   比如：
   - 强制关闭 intent
   - 强制过期
   - operator cancel
   - 冻结某个 symbol 新开仓
   - 强制 reconcile-only

别把这两个混了。


十三、OMS 是干什么的，用最简单的话讲

OMS 不是研究模块。
OMS 也不是打分模块。
OMS 是“执行真相中心”。

它管的是：
   - 真实账户
   - 真实持仓
   - 真实订单
   - 真实成交
   - desired 和 actual 的差距
   - 未完成 intent 如何跨天继续

一句话：
研究层负责“我想买什么”
OMS 负责“我实际上买成了什么”


十四、你什么时候应该用 oms_validate

1. 你改了 OMS 代码
2. 你改了执行桥
3. 你改了 intent continuity
4. 你改了 manual override
5. 你怀疑 OMS 状态机坏了

它不是实盘命令。
它是执行侧的回归检查命令。


十五、如果你只是想“最稳地用”

最稳妥流程就是：

1. 只从 launch_canonical.py 进
2. 夜里跑 research_only
3. 必要时单独跑 release_only
4. 白天先 gate-only
5. 确认安全、账户、时间都没问题后，再 execution_only
6. 精准模式下，没决定交易前，precision_trade 默认保持 off


十六、最容易犯的几个错误

1. 直接拿 integrated_supervisor 当正式生产入口
2. 用错 Python 环境去碰 gmtrade
3. 手改自动生成 config
4. 忘了 release_only，以为研究结果会自动进入执行层
5. 忘了看 safety / OMS，就以为“程序没反应”
6. 把 trade_clock 的 override 和 OMS 的 override 混在一起


十七、给你一句最终记忆版

平时你只要记住：

白天交易靠 execution_only
夜里研究靠 research_only
正式衔接靠 release_only
一直盯时间的是 trade_clock_service
真正的执行真相归 OMS
能不能动先看 safety

这六句话就够你日常用系统了。
