系统日常使用说明（稳定运行版）

一、先用一句话理解系统

这套系统现在分成 4 层：

1. 研究层
   负责夜里做研究，产出目标组合。

2. 发布层
   负责把目标组合冻结成正式 release。

3. 执行层
   负责在交易窗口读取 release，然后进入 OMS / execution 去做执行。

4. 调度层
   负责按时间自动拉起研究、发布、盘前门禁、上午执行、午间复盘、下午执行、日报。
   自动 shadow 当前默认关闭。

你平时不要把它理解成“一个脚本一次跑完全部事情”。
正确理解是：

研究负责“想”
发布负责“定稿”
执行负责“做”
调度层负责“按时间拉起来并留痕”

四、现在推荐的研究档位

目前有 3 个 profile：

1. quick_test
   用于轻量联调。
   V5 cycles = 1。

2. daily_production
   用于日常自动主链。
   V5 cycles = 3。
   这是稳定运行测试阶段的默认自动研究档。

3. overnight
   用于更重的夜间研究。
   V5 cycles = 8。
   现在不再作为日常自动主链默认档。

为什么日常自动链不用 overnight：

因为 overnight 太重，更适合“重研究”，不适合“每天稳定准时产出”。
现在自动主链追求的是：

- 准时
- 稳定
- 有 fallback
- 不拖跨第二天执行

所以自动主链默认用 daily_production，不用 overnight。


五、最常用的 mode 是什么

1. research_only
   跑正式研究链。
   会走到目标组合产出。

2. release_only
   把当前最新组合建议发布成正式 release。

3. execution_only
   只做执行，不重跑研究。

4. oms_validate
   只做 OMS 硬化验证，不跑重研究链。

5. integrated_supervisor
   仍然存在，但现在主要用于联调。
   不建议作为稳定运行测试阶段的正式日常主流程。


六、调度器每天自动跑什么

调度器是手动启动、持续运行的轻进程。
它自己不直接做重研究，只会按 phase 拉起子进程。

当前自动 phase 是：

1. research
   建议时间：15:05
   命令逻辑：research_only + daily_production
   目标：为下一交易日准备研究结果

2. release
   建议时间：15:10
   命令逻辑：release_only
   目标：冻结正式 release

3. preopen_gate
   建议时间：09:20
   命令逻辑：execution_only --gate-only --ignore-window
   目标：盘前检查今天有没有资格尝试执行

4. simulation
   建议时间：09:30:35
   命令逻辑：execution_only，走 simulation namespace
   目标：跑拟真实执行链，暴露实际落地问题

5. midday_review
   建议时间：11:35
   命令逻辑：midday_review_only
   目标：读取上午 OMS 真相，判断下午是否继续执行、用哪个 namespace、是否允许 unfinished-order reconcile

6. afternoon_execution
   建议时间：13:05
   命令逻辑：execution_only，沿用上午真实 namespace
   目标：做下午二次调整，不重新跑研究链

7. summary
   建议时间：15:20
   目标：把今天的 phase、release、OMS、warnings 整理成日报和运行包


七、simulation 和 shadow 的区别

1. simulation
   是主执行观测线。
   会进入 execution / OMS / audit。
   用来观察系统真正落地时会发生什么。

2. shadow
   当前不要把它理解成“完全无风险的只读模拟线”。
   现阶段 shadow_run 还没有 broker-isolated 的真 shadow 语义。
   也就是说，shadow 现在只是 namespace 隔离，不等于 broker 隔离。
   所以自动 shadow 调度目前默认关闭。
   在没有专门加 broker no-submit 保护前，不建议盘中手动开 shadow。

它们会吃同一版 release。
但它们不会共用一套 OMS 账本。

当前隔离方式是：

- simulation OMS: ...\oms_v1\simulation\...
- shadow OMS: ...\oms_v1\shadow\...

所以不要担心 shadow 污染 simulation，或 simulation 污染 shadow。


八、研究失败时会发生什么

现在不是“研究失败，第二天所有东西一起崩”。
调度器已经有 fallback 规则。

大意是：

1. research 超时或失败
   不会无限等。

2. release 阶段会尝试 fallback
   优先使用最近一版仍然可接受的组合产物，再重新发布一版新的正式 release。

3. 如果 fallback 也不可用
   simulation 会跳过。
   shadow / gate / 日报仍会继续按规则留下痕迹。

所以现在的原则是：

- 不无限拖延
- 不让后续 phase 叠在一起
- 失败要留痕
- 能降级就降级，不能降级就明确 skip


九、如何手动启动调度器

最常用命令：

powershell -ExecutionPolicy Bypass -File F:\quant_data\Ashare\scripts\start_trade_clock.ps1 -Profile daily_production

如果你只想前台看输出，可以加：

powershell -ExecutionPolicy Bypass -File F:\quant_data\Ashare\scripts\start_trade_clock.ps1 -Profile daily_production -Foreground

说明：

- 默认不会开机自启
- 默认不会自动安装计划任务
- 这是一个手动启动的长期运行进程
- 它默认走后台子进程 + stdout/stderr 落盘，目的就是绕开 IDE / ACP 前台长命令不稳定的问题


十、如何手动停止调度器

最常用命令：

powershell -ExecutionPolicy Bypass -File F:\quant_data\Ashare\scripts\stop_trade_clock.ps1

它的行为是：

1. 写 stop request 文件
2. 调度器在空闲等待或下一次 stop 检查时优雅退出
3. 不会乱删 OMS / execution 真相账本

如果你真的要立即停掉，可以用：

powershell -ExecutionPolicy Bypass -File F:\quant_data\Ashare\scripts\stop_trade_clock.ps1 -Force

但正常情况下不建议先上 Force。


十一、如何手动看当前状态

这个脚本会把下面这些东西用比较直观的方式打印出来：

十三、每天最应该看哪些文件

1. 调度器心跳
   F:\quant_data\Ashare\data\trade_clock\clock_state.json

   这个文件的意思：

   - 服务是否还活着
   - 现在是不是 stop_requested
   - 下一次应该跑哪个 phase
   - 当前 gate / safety 的高层状态

2. 调度器运行状态
   F:\quant_data\Ashare\data\trade_clock\runtime\scheduler_runtime.json

   这个文件的意思：

   - 调度器自己的内部运行态
   - active_phase
   - stopping / starting / running
   - 最后更新时间

3. 当天 / 当期 phase 状态
   F:\quant_data\Ashare\data\trade_clock\phase_state\YYYYMMDD.json

   这个文件的意思：

   - 今天每个 phase 到底有没有跑
   - 是 success / failed / skipped / timeout 哪一种
   - stdout/stderr 落在哪个文件

4. 最新正式 release 指针
   F:\quant_data\Ashare\data\trade_release_v1\latest_release.json

   这个文件的意思：

   - 当前执行层真正会去读哪一版 release
   - trade_date 是哪天
   - source_mode 是正常 release 还是 fallback_release

5. 安全层真相
   F:\quant_data\Ashare\data\trade_clock\system_safety_state.json

   这个文件的意思：

   - 当前系统是不是 NORMAL / CAUTION / PANIC / HALT
   - halt_reason 是什么
   - 是安全层不让动，还是调度器自己没跑到

6. simulation OMS 真相
   F:\quant_data\Ashare\data\live_execution_bridge\oms_v1\simulation\snapshots\oms_summary.json

   这个文件的意思：

   - simulation 那条线实际上发生了什么
   - 有没有 dispatch
   - 有没有未完成 intent
   - gap / actual_state 是否正常

7. shadow OMS 真相
   F:\quant_data\Ashare\data\live_execution_bridge\oms_v1\shadow\snapshots\oms_summary.json

   这个文件的意思：

   - shadow 理论基线那条线的 OMS 留痕
   - 用来和 simulation 对照

8. 每日自动运行包
   F:\quant_data\Ashare\outputs\automation_runs\YYYYMMDD\

这个目录里最重要的是：

- run_manifest.json
- phase_status.json
- daily_release_summary.json
- market_state_summary.json
- portfolio_v2a_summary.json
- oms_summary_simulation.json
- oms_summary_shadow.json
- warnings.json
- critical_flags.json
- daily_report.txt

   这个目录的意思：

   - 这是“今天这一整天自动链最后留给人看的包”
   - 如果你不想翻很多散文件，优先看这里


十四、如果程序“没动”，先按什么顺序查

按这个顺序最有效：

1. 先看 clock_state.json
   看 service_alive / next_due_phase / stop_requested

2. 再看 phase_state\YYYYMMDD.json
   看卡在哪个 phase

3. 再看 latest_release.json
   看今天到底有没有正式 release

4. 再看 system_safety_state.json
   看是不是 safety 把系统拦住了

5. 再看 simulation / shadow 的 OMS summary
   看是不是 gap、continuity、dispatch 出了问题

6. 最后再去看每个 phase 的 stdout/stderr 日志
   这些日志会被收进 outputs\automation_runs\YYYYMMDD\logs\ 里

十六、最简单的记忆版

你只要记住下面 8 句就够了：

1. 日常自动研究档现在是 daily_production，不是 overnight。
2. 调度器必须手动启动，不做开机自启。
3. 调度器只负责按时间拉 phase，不直接跑重研究逻辑。
4. research 负责出目标组合。
5. release 负责冻结正式版本。
6. simulation 是主执行观测线。
7. midday_review 负责决定下午要不要继续动手。
8. 当前自动 shadow 默认关闭，出问题先看 clock_state，再看 phase_state，再看 release / safety / OMS。
